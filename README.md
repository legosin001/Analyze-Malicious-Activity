## 📋 Project Overview

This repository documents a **hands-on cybersecurity lab** focused on analysing malicious network and application activity. Working across a multi-machine lab environment (Windows Server 2022 and Kali Purple), I simulated real-world attacks and observed how they appear from a defender's perspective — using tools like Event Viewer, netstat, and Task Manager.

The lab covered three attack categories: **password/brute force attacks**, **command injection with reverse shell**, and **SYN flood DoS attacks**.

> 📅 **Completed:** April 2026  
> 🖥️ **Environment:** Windows Server 2022 (Domain Controller + Member Server) + Kali Purple 2023.1

---

## 🎯 What I Learned

- How attackers enumerate SMB services using `nmap` and `enum4linux`
- How a scripted brute force attack generates detectable patterns in Windows Event Viewer
- How command injection vulnerabilities work and how to exploit them
- How to establish a **reverse shell** through a web application vulnerability
- What normal vs. attack-state network traffic looks like in Task Manager and `netstat`
- How a **SYN flood DoS attack** overwhelms a server's TCP handshake process

## 🧪 Exercises Completed

### Exercise 1 — Observe Indications of a Brute Force Attack

| Task | Description |
|------|-------------|
| Task 1 | Perform SMB enumeration using `nmap` and `enum4linux` from Kali, then attempt an SMB login |
| Task 2 | Run a scripted bash brute force attack (`brute.sh`) and observe the flood of errors in Windows Event Viewer Security logs |

### Exercise 2 — Conduct Command Injection and Observe Indications

| Task | Description |
|------|-------------|
| Task 1 | Exploit the BWAPP web application's OS Command Injection vulnerability to run system commands on the Windows host |
| Task 2 | Use `ncat` to establish a **reverse shell** from the Windows victim machine back to Kali, then verify the active connection with `netstat` |

### Exercise 3 — Observe Indications of a SYN Flood Attack

| Task | Description |
|------|-------------|
| Task 1 | Observe normal baseline network activity in Windows Task Manager (Ethernet performance) and `netstat` |
| Task 2 | Launch a SYN flood attack using Metasploit's `auxiliary/dos/tcp/synflood` module and watch the victim's network activity spike — observing `SYN_RECEIVED` connections pile up |

## 🛠️ Tools Used

| Tool | Purpose |
|------|---------|
| `nmap` | Port scanning and service discovery |
| `enum4linux` | SMB enumeration — usernames, shares, policies |
| `smbclient` | SMB connection and login attempts |
| `brute.sh` | Bash script for automated brute force login attempts |
| Windows Event Viewer | Detecting and reviewing security log errors |
| BWAPP | Intentionally vulnerable web application for command injection |
| `ncat` | Network communication tool — used to establish reverse shell |
| Metasploit Framework | `auxiliary/dos/tcp/synflood` module for SYN flood attack |
| `netstat` | Observing active connections and TCP states |
| Windows Task Manager | Monitoring network throughput during DoS attack |

## ❓ Knowledge Check — Can You Answer These?

These are questions anyone who works through this lab should be able to answer. Test yourself before revealing the answers!

| Q | Question |
|---|----------|
| 1 | Which two TCP ports are associated with the SMB protocol? |
| 2 | What Windows tool would you check to observe security log errors from a brute force attack? |
| 3 | What command separator allowed OS command injection to succeed in this lab? |
| 4 | What does a reverse shell mean in terms of who initiates the connection? |
| 5 | In a SYN flood attack, what TCP handshake state fills up on the victim's machine? |

<details>
<summary>✅ Click to reveal answers</summary>

| Q | Answer |
|---|--------|
| 1 | **Ports 139 and 445** |
| 2 | **Windows Event Viewer** — specifically the SMBServer > Security logs |
| 3 | The **pipe `\|`** character worked as a command separator |
| 4 | The **victim initiates** the connection back to the attacker's machine |
| 5 | **SYN_RECEIVED** — the server receives SYN packets but the attacker never completes the handshake |

</details>
